# Project IronLog — Threat Model

> **Document Classification:** CONFIDENTIAL — Threat Intelligence  
> **Methodology:** STRIDE per-component + MITRE ATT&CK Mapping  
> **Scope:** Core IronLog infrastructure and endpoint agents

---

## Executive Summary

This document outlines the top 5 high-priority threat vectors against the Project IronLog architecture. Each vector assumes an advanced persistent threat (APT) capable of localized infrastructure compromise. Mitigations are layered across Prevention, Detection, and Response.

---

## Threat Vector 1: Kernel Rootkit Bypassing eBPF

**Description:** An attacker gains `CAP_SYS_ADMIN` privileges on a monitored endpoint, loads a malicious kernel module (LKM), and hooks kernel functions below the VFS layer to hide file modifications from the IronLog eBPF agent.

- **STRIDE Category:** Elevation of Privilege / Tampering / Evasion
- **MITRE ATT&CK ID:** T1014 (Rootkit), T1562.001 (Disable or Modify Tools)
- **CVSS v3.1 Severity:** **HIGH (8.2)** `CVSS:3.1/AV:L/AC:L/PR:H/UI:N/S:C/C:N/I:H/A:H`
- **Detection Gap:** If the rootkit successfully hooks `vfs_write` prior to the eBPF tracepoint, the agent will never receive the file-change event, creating a blind spot.

### Layered Mitigation
1. **Prevention:** 
   - Enforce UEFI Secure Boot and kernel lockdown mode (`lockdown=confidentiality`) to prevent unsigned LKM loading.
   - Disable `kexec` and restrict `/dev/kmem` access.
2. **Detection:**
   - **Out-of-band verification:** The AI Anomaly Detection layer identifies anomalies in process lineage (e.g., unexpected privilege escalation chains) even if specific file writes are masked.
   - Run periodic remote memory forensics via the SIEM to detect hidden LKM data structures.
3. **Response:**
   - SOAR playbook automatically quarantines the endpoint at the network switch level upon detection of kernel-space tampering.
   - Revoke the agent's identity in the Hyperledger Fabric MSP, preventing any further ledger commits from the compromised node.

## Threat Vector 2: IPFS Cluster DDoS & CID Poisoning

**Description:** An adversary floods the private IPFS cluster with massive garbage payloads (DDoS) or attempts to submit a malicious payload resulting in a CID collision (Poisoning) to overwrite or obscure legitimate audit evidence.

- **STRIDE Category:** Denial of Service / Spoofing
- **MITRE ATT&CK ID:** T1498 (Network Denial of Service), T1565.001 (Stored Data Manipulation)
- **CVSS v3.1 Severity:** **MEDIUM (6.5)** `CVSS:3.1/AV:N/AC:L/PR:L/UI:N/S:U/C:N/I:L/A:H`
- **Detection Gap:** The ledger records the CID, but if the IPFS cluster is overwhelmed, retrieving the actual forensic payload for analysis will timeout, delaying incident response.

### Layered Mitigation
1. **Prevention:**
   - Implement network-level rate limiting (e.g., via Envoy proxy) in front of the IPFS cluster.
   - The IronLog agent locally rate-limits (token bucket: 1000 events/sec) to prevent single-endpoint event storms.
   - **Poisoning defense:** IPFS uses cryptographic hashing (SHA-256 by default) for content addressing. A targeted CID collision requires breaking SHA-256 preimage resistance, which is computationally infeasible.
2. **Detection:**
   - Monitor IPFS cluster `ipfs-cluster-service` metrics (pin queue length, disk I/O, network ingress) via Prometheus. Alert on anomalies exceeding 3-sigma deviation.
3. **Response:**
   - Auto-scale IPFS read/write nodes based on queue depth.
   - If IPFS is unavailable, IronLog agents fallback to local encrypted buffering (max 10,000 events) and replay when cluster health is restored.

## Threat Vector 3: Compromised HSM Key Material (Side-Channel)

**Description:** An attacker with physical or adjacent network access to the HSM appliance exploits a zero-day side-channel vulnerability (e.g., timing attack, power analysis) to extract the ECDSA-P384 private key used by an endpoint agent.

- **STRIDE Category:** Information Disclosure / Spoofing / Repudiation
- **MITRE ATT&CK ID:** T1552.004 (Private Keys), T1588.004 (Digital Certificates)
- **CVSS v3.1 Severity:** **CRITICAL (9.0)** `CVSS:3.1/AV:A/AC:H/PR:N/UI:N/S:C/C:H/I:H/A:N`
- **Detection Gap:** Signatures generated by the extracted key are cryptographically valid. The blockchain smart contract cannot distinguish them from legitimate agent activity.

### Layered Mitigation
1. **Prevention:**
   - Utilize only FIPS 140-2 Level 3 (or Level 4) certified HSMs, which mandate physical tamper-response mechanisms (zeroization upon physical intrusion).
   - Require strictly authenticated, TLS-encrypted (PKCS#11 over TLS) API access to the HSM.
2. **Detection:**
   - **AI Layer:** The AI anomaly detection engine will likely flag the *behavior* of the fabricated logs. For example, if an attacker uses the stolen key to inject logs from an unexpected IP address, or creates anomalous `cadence_delta` patterns.
   - HSM audit logging (forwarded directly to SIEM, bypassing IronLog) alerts on anomalous key export attempts or excessive signing rates.
3. **Response:**
   - Immediately trigger key revocation via the Fabric chaincode (updating the allowed MSP list).
   - The HSM is logically isolated, and a root key ceremony is initiated to rotate the master partition keys.

## Threat Vector 4: Rogue Blockchain Node (Supply Chain)

**Description:** An attacker compromises the software supply chain of one of the organizations hosting a Hyperledger Fabric peer or orderer node. The attacker deploys a modified node binary attempting to drop, reorder, or forge ledger transactions.

- **STRIDE Category:** Tampering / Repudiation
- **MITRE ATT&CK ID:** T1195.002 (Compromise Software Supply Chain)
- **CVSS v3.1 Severity:** **HIGH (7.7)** `CVSS:3.1/AV:N/AC:L/PR:H/UI:N/S:C/C:N/I:H/A:N`
- **Detection Gap:** The compromised node acts as an insider threat within the ledger network.

### Layered Mitigation
1. **Prevention:**
   - **Endorsement Policy:** Require 3-of-5 organizations to endorse any transaction. A single rogue node cannot unilaterally commit state changes.
   - **Consensus:** The Raft ordering service requires a strict majority. 1 or 2 compromised orderers out of 5 cannot manipulate block ordering.
   - Strict mTLS authentication between all network components.
2. **Detection:**
   - Cross-org gossip protocol monitoring: Benign peers will detect and log endorsement mismatch failures if the rogue node attempts to return tampered Read/Write sets.
   - SIEM monitors Fabric peer logs for `ENDORSEMENT_MISMATCH` and `MVCC_READ_CONFLICT` errors.
3. **Response:**
   - Org administrators vote to evict the compromised organization's MSP from the channel configuration.
   - Re-deploy the node from a known-good, deterministically reproducible source build.

## Threat Vector 5: AI Model Adversarial Poisoning

**Description:** A malicious insider (e.g., a compromised ML engineer account) subtly alters the training dataset or the ONNX model weights before deployment to introduce a deliberate blind spot (e.g., ignoring logs originating from a specific attacker-controlled process path).

- **STRIDE Category:** Tampering / Information Disclosure
- **MITRE ATT&CK ID:** T1565.001 (Stored Data Manipulation - Model Poisoning)
- **CVSS v3.1 Severity:** **HIGH (8.5)** `CVSS:3.1/AV:N/AC:H/PR:L/UI:N/S:C/C:N/I:H/A:H`
- **Detection Gap:** The model functions normally for 99.9% of traffic, making the backdoor extremely difficult to detect via standard accuracy/recall metrics.

### Layered Mitigation
1. **Prevention:**
   - **M of N Approval:** Model weight deployment requires 2-of-3 ML engineer cryptographic signatures.
   - The hash of the approved ONNX model is committed to the Fabric blockchain. The AI inference engine validates its local model hash against the ledger at startup.
2. **Detection:**
   - **Continuous Verification:** The SIEM periodically injects synthetic "red team" anomaly logs into the Kafka stream. If the AI engine fails to flag these known anomalies (a "canary" failure), an immediate critical alert is generated.
   - Model explainability outputs (e.g., SHAP values) are logged. Sudden shifts in feature importance distributions are flagged.
3. **Response:**
   - The SOAR playbook automatically rolls back the AI engine to the previous known-good model hash from the artifact registry.
   - Forensic lockdown of the CI/CD pipeline used for model training to identify the poisoning source.
